# 16. SOP Graphics, KPI and Redraft Quality Gate

> Version: v1 | Date: 2026-03-27 | Owner: QMS Engineer

---

## 1. Purpose

This document covers the operational lessons and root errors encountered during the SOP standardization on 2026-03-27 to:

- Does not recur flowchart error with only one bubble color remaining,
- Do not repeat KPIs that are written for the sake of it, have no operational value,
- Do not repeat editorial notes in the SOP body,
- Do not repeat mechanical version increases or leave old versions in unreleased documents,
- Do not re-update Section 6 / 7 according to a mechanical template without following the actual SOP logic.

This document is required reading when:

- create new SOP,
- rewrite Section 6,
- rewrite Section 7,
- edit generator/script to generate SOP,
- Review quality before considering an SOP complete.

---

## 2. Original lessons must be locked

### 2.1 Flowchart graphic error has occurred

Error encountered:

- `proc-num` in the detailed step has the correct rotation color,
- but `flow-num` in the flowchart falls to a default color or does not have a corresponding balloon color.

The root cause can come from one or more points:

1. Generator does not generate inline styles for each `flow-step` / `flow-num` / `flow-arrow`.
2. CSS fallback has no or incorrectly written child position logic.
3. Flowchart has `.flow-arrow` in between `.flow-step`, so if you naively choose `nth-child` you will map the wrong color.
4. `.active` or `.critical` overrides the step's color bubble instead of just supporting border/background.

### 2.2 KPI errors have occurred

Error encountered:

- KPI is only written as `được kiểm soát`, `đúng hạn`, `đủ hồ sơ`,
- KPIs do not have numerical thresholds,
- KPI has no data source,
- KPI has no reaction trigger,
- KPI copied from another family without following the exact nature of that gate.

### 2.3 Hygiene errors have occurred

Error encountered:

- notes like `Bổ sung theo note`, `Liên kết note`, `Quy tắc dùng thuật ngữ`, `so với bản trước` are included in the SOP,
- missing header `V1` while the document has not yet been released,
- rationale benchmark or editorial notes are mixed into the body of operational documentation.

### 2.4 Normalized boundaries must be locked

Normalization is allowed to focus on `core-standard`:

- color palette,
- CSS fallback,
- HTML structure,
- technical QA checklist,
- format KPI table / flowchart / section.

Do not use this as a reason to upgrade `nội dung SOP` in bulk.

Required rules:

- Graphics and formatting can be edited at the core and then applied to multiple documents,
- KPI content, gate wording, step wording, purpose, scope, role, input/output and exception must be researched according to each SOP.

## 3. Flowchart graphic standards must be maintained

### 3.1 Two layers of protection required

Proper flowchart graphics must be protected by both layers:

1. **Layer 1: inline style generated by generator**
2. **Layer 2: Global CSS fallback in `assets/style.css`**

Don't rely on just one class.

### 3.2 Layer 1: rules for generator/new generated HTML

With new SOP generated by script, each flowchart step should have:

- `flow-step` has inline `border-color` and `background`
- `flow-num` has inline `background:linear-gradient(...)`
- `flow-arrow` has inline `color`

Minimal example:

```html
<div class="flow-step" style="border-color:rgba(21,101,192,0.28);background:linear-gradient(135deg,rgba(21,101,192,0.10) 0%, rgba(255,255,255,0.98) 64%);">
  <div class="flow-num" style="background:linear-gradient(135deg,#1565c0,#1976d2)">1</div>
  <div class="flow-text"><div class="flow-title">Tên bước</div></div>
</div>
<div class="flow-arrow" style="color:rgba(25,118,210,0.45)">→</div>
```

### 3.3 Layer 2: rules for CSS fallback

Right CSS fallback:

- applied to `.flowchart > .flow-step`,
- color map according to the actual child position of `.flow-step`,
- Remember that `.flow-arrow` is in between steps.

Therefore, fallback selectors often have to follow the pattern:

- `20n+1`
- `20n+3`
- `20n+5`
- ...

Do not write fallback as if every child in `.flowchart` is a step.

### 3.4 Relations with `.active` and `.critical`

`.active` and `.critical` are just for:

- press border,
- press block background,
- adds visual semantics to the decision/test step.

Do not use `.active` or `.critical` to lose:

- real serial number,
- true color mapping between flowchart and `proc-num`,
- palette consistency according to step index.

### 3.5 Standard Palettes

Current standard rotating palette:

| Step | Gradient |
|---|---|
| 1 | `#1565c0 → #1976d2` |
| 2 | `#059669 → #10b981` |

### 3.6 Minimum graphics test

Before considering the flowchart as standard, you must confirm:

1. `flow-num count = proc-num count`
2. The number of steps increases continuously `1...n`
3. Each `flow-num` maps to exactly one `proc-num`
4. Automatically generated SOP has inline style on `flow-num`
5. Old files/handwritten files still have color thanks to CSS fallback
6. No more SOPs falling back to a default color just because of a missing inline style

## 4. KPI Section 6 standards must be maintained

### 4.1 Each IG must answer all 5 questions

Each IG row must fully answer:

1. What is the use of keeping this gate?
2. Who has the right to keep/open the gate.
3. When is HOLD required?
4. Measure gate effectiveness using any index.
5. If there is a deviation, how should the system react?

### 4.2 Standard KPI formula

A standard KPI statement should read as follows:

`metric + threshold + source or operating context + reaction trigger`

For example:

- `100% contract review hoàn tất trước commit; mismatch sau commit = 0; ACK thay đổi khách hàng <= 1 ngày làm việc.`
- `Backup success >= 99%; restore test dữ liệu critical = 100% theo quý; failed restore không có action = 0.`
- `100% tín hiệu out-of-control phản ứng trước lot kế tiếp hoặc <= 1 giờ; đặc tính trọng yếu giữ Cpk/Ppk >= 1.33 hoặc có reaction plan được duyệt.`

### 4.3 KPI does not meet standards if

- no number,
- no SLAs,
- no measuring source,
- no trigger response,
- does not speak to the effectiveness of the gate,
- Just a beautiful description.

Non-standard example:

- `được kiểm soát`
- `đúng hạn`
- `đủ hồ sơ`
- `cải thiện liên tục`
- `đảm bảo an toàn dữ liệu`

### 4.4 Base on closing KPI numbers

Each KPI should be pegged from one of three sources:

1. **Hard requirements**: laws, standards, customer requirements, technical standards
2. **Official benchmark**: APQC, Minitab, NIST, official ERP/MES/QMS documentation, or equivalent technical source
3. **Inferred internal target**: stricter than benchmark because of risk or HESEM's operation

If it is an internal deductive target, the rationale must be written in the working note or draft document, not in the SOP body.

### 4.5 KPI family starting point to remember

The numbers below are practical starting points, not hard rules for all SOPs:

| Family | Starting point should be considered |
|---|---|
| Contract review / ACK | `ACK <= 1 ngày làm việc`, `mismatch sau commit = 0` |
| Engineering release | `100% có approver + evidence`, `release sai revision = 0` |
| Receiving / material readiness | `dock-to-ready critical <= 24 giờ`, `supplier document escape = 0` |
| Planning/dispatch | `schedule attainment >= 90%` |
| First-piece / FAI | `pass ngay lần đầu >= 95%` |
| Production restart / hold | `restart không re-authorization = 0` |
| Final release / shipping | `document accuracy >= 99.5%`, `thiếu chứng từ = 0` |
| Invoice/closeout | `first-time-right invoicing >= 98%`, `invoice <= 1 ngày làm việc` |
| Access control | `cấp/đổi/thu hồi quyền <= 1 ngày làm việc`, `orphan account = 0` |
| Backup / restore | `backup success >= 99%`, `restore test = 100% theo chu kỳ` |
| Records / retention / disposal | `duplicate live record = 0`, `sanitization compliance = 100%` |
| MSA/capability | `GRR < 10%` is good, `10–30%` is only used conditionally, `Cpk/Ppk >= 1.33` is a common reference threshold |

### 4.6 KPI families cannot be copied horizontally

Do not take KPIs from one family and mechanically assign them to another family.

Wrong example:

- get setup-machine KPI assigned to `Retention và secure disposal`
- Take the finance KPI and assign it to gate `learn-back` just because it has the word `close`
- get KPI FAI because substring `fai` accidentally matches from `failure`

So every script map title → KPI family must:

- prioritize token-aware matching,
- don't use naive substring,
- Have an audit after giving birth.

## 5. Mandatory hygiene for unreleased SOP

### 5.1 Version

- Unreleased documents always retain `V0`.
- Do not raise `V1`, `V1.x`, `V2` just because the draft has gone through many rounds of editing.

### 5.2 What should not appear in body

- `Bổ sung theo note`
- `Liên kết note`
- `Quy tắc dùng thuật ngữ`
- `so với bản trước`
- `điểm mới`
- notes benchmark
- note reasoning of AI
- notes editorial / implementation

### 5.3 Where drafting traces are allowed to be kept

If you need to save:

- benchmark sources,
- Reasons for choosing KPI number,
- difference between old draft and new draft,
- mapping logic of the script,

then save it at:

- DCR,
- review logs,
- working memo,
- core-standard,
- generator profile,
- commit history.

Do not save in the operating SOP body.

## 6. Quality gate when rewriting Section 6 / 7

### 6.1 Before repair

Must complete:

1. read old documents,
2. determine what operational logic must be kept,
3. Research suitable external sources,
4. pin the IG number according to control architecture,
5. lock the number of steps according to the operating sequence.

### 6.2 When editing Section 6

- just replace the part between `p6` and `p7`,
- Do not delete heading `p6` by mistake,
- Do not delete heading `p7` by mistake,
- Delete all old logic if it is wrong, do not "patch" half-heartedly.

### 6.3 When editing Section 7

- just replace the part between `p7` and `p8`,
- Do not delete heading `p7` by mistake,
- Do not delete heading `p8` by mistake,
- If the old flow and detailed steps are no longer correct, delete and rewrite entire Section 7.

### 6.4 After repair

Must check:

1. `flowchart steps = detailed steps`
2. correct color bubble
3. True numeric KPIs
4. No more editorial notes
5. header version is correct `V0` if not yet issued
6. Section 6 and Section 7 are not mechanically updated according to the template

---

## 7. Requirements with generator / script

Generator creating SOP must not just "generate HTML", but must have minimal self-checking:

1. check `flow-num count = proc-num count`
2. Check all new `flow-num` have inline style
3. Check that there are no longer any banned editorial chains
4. Check KPIs of IG rows with numbers or SLA
5. Check that the mapping title family is no longer `NONE`
6. Use token-aware matching, not naive substrings for family classification

If a generator does not pass these tests, it cannot be used for mass regeneration.

---

## 8. Final checklist for writers or QA

- [ ] Header version is correct `V0` if the document has not been released for the first time
- [ ] No more editing notes in the body
- [ ] No more benchmark notes / terminology notes in the body
- [ ] Flowchart bubble has the correct palette
- [ ] Flowchart does not fall into a single color
- [ ] Flowchart steps = detailed steps
- [ ] KPIs of each IG have numbers or real SLA
- [ ] KPI speaks to gate effectiveness, not just a description
- [ ] KPI family matches SOP logic, not copied horizontally
- [ ] Section 6 and Section 7 stick to old documents + research, not mechanical work
