# Document Graphics Governance 2026-04-05

## Purpose

This document defines the non-negotiable graphics and UI governance model for the document platform inside HESEM QMS.

Scope:

- controlled documents
- evidence control / online forms
- evidence vault
- compliance reports
- eQMS runtime
- schema studio data views
- document admin panels

The goal is not "visual consistency when convenient".

The goal is:

1. one source of truth
2. zero guessing for future teams
3. zero hidden hardcode paths
4. near-total consistency across ERP, MES, and eQMS modules

This standard assumes greenfield discipline. Legacy modules may use temporary bridge layers, but new modules must start from this contract directly.

## Source Of Truth Stack

All document-facing UI must follow this exact order:

1. `Admin Appearance` is the governed control surface for proposed design tokens and governed patterns; backend graphics authority owns the production version/ETag and publish state.
2. `Theme runtime` maps saved configuration to CSS variables.
3. `Shared tokens` expose semantic values such as surfaces, text, border, radius, sizing, density, and motion.
4. `Shared components` consume those tokens through `hm-*` primitives or equivalent shared selectors.
5. `Bridge aliases` may translate old module-local variables back to the shared token layer.
6. `Module UI` may arrange layout and content, but may not invent a separate visual system.

Important:

- Layers 5 and 6 must always depend on layers 1-4.
- No lower layer may override the authority of a higher layer.

## Mandatory Rules

### 1. No private design systems

Modules may not define autonomous systems such as `--ec-*`, `--eqms-*`, `--ev-*`, `--cr-*` unless those variables are only aliases back to shared tokens.

Allowed:

```css
.module-root {
  --ev-surface: var(--bg-surface);
  --ev-border: var(--border);
}
```

Not allowed:

```css
.module-root {
  --ev-surface: #ffffff;
  --ev-border: #d0d7e2;
}
```

### 2. No JS-generated visual shell

JavaScript must not inject the main stylesheet of a module when a normal stylesheet can exist in `/styles`.

Allowed:

- geometry calculated from data
- transient fallback for emergency load failure

Not allowed:

- building tabs, cards, forms, badges, tables, spacing, shadows, and color system inside a `style` tag generated by JS

### 3. Inline style is not a design system

Inline style is allowed only for:

- data-dependent width or height
- transform or position derived from runtime values
- SVG geometry

Inline style is not allowed for reusable design decisions such as:

- control padding
- corner radius
- border color
- typography scale
- surface colors
- hover / active / selected states

### 4. Every governed token needs three things

No new Admin Appearance option may be added unless all three are present:

1. a real runtime mapping
2. a visual preview inside Admin Appearance
3. at least one real consumer in the product

If any of the three is missing, the option is not ready.

### 5. Every shared component must support all global modes

At minimum:

- light
- dark
- auto / OS
- scheduled theme
- density response
- focus-visible
- disabled
- empty state

### 6. Zero hidden floors

If a token allows `0`, the rendered UI must actually reach `0` unless accessibility rules explicitly require a minimum target.

Typical hidden-floor bugs:

- `min-height`
- native `appearance`
- inherited `line-height`
- hardcoded `padding`
- legacy module wrapper CSS

### 7. Review rejection policy

Code review must reject any new UI if:

- a designer or developer cannot tell which Admin Appearance control affects it
- the component ignores light/dark
- the component ignores shared density
- the module introduces private hardcoded semantic colors

## Approved Build Pattern For New Modules

New ERP/MES/eQMS modules must follow this structure:

1. root module class
2. shared token alias layer only if needed
3. shared component classes wherever possible
4. module-specific layout classes only for composition

Example:

```css
.mes-workspace {
  --module-surface: var(--bg-surface);
  --module-border: var(--border);
}

.mes-toolbar .hm-btn { }
.mes-filter .hm-input { }
.mes-results .hm-table { }
```

## Admin Appearance Contract

Admin Appearance now serves two roles:

1. token editor
2. governance console

Every future graphics-related improvement must be represented in Admin Appearance only when:

- it is reusable across multiple screens or products
- it expresses a real design token or governed pattern
- it can be previewed in isolation
- it can be verified in at least one live module

Do not add Admin Appearance controls for one-off page decoration.

## Forbidden Anti-Patterns

- module-only hardcoded light palette
- module-only hardcoded dark palette
- button height controlled by one screen and padding controlled by another
- preview that changes while the live module does not
- config stored but not consumed
- consumer exists but is unreachable from Admin Appearance
- runtime mapping that writes base semantic vars in the wrong mode
- fallback modal or overlay that ignores tokens

## Release Gate Checklist

Every new module or major UI refactor must pass:

- shared stylesheet exists in `/styles`
- no JS-generated visual shell
- zero orphan tokens
- zero unreachable controls
- backend graphics compliance matrix reviewed
- impact analysis recorded for token/theme/component/template changes
- release blocker snapshot attached
- waiver validity checked when a module is not full-admin-controlled
- one real live consumer for each new governed token
- light/dark parity checked
- density floor checked
- WCAG AA text contrast checked
- preview matches real module behavior
- governance notes updated

## Graphics Control Plane Upgrade - 2026-04-13

Admin Appearance is now governed as the Graphics Control Plane. It is not only a theme editor. The control path is:

`Admin Appearance -> backend graphics authority -> controlled template registry -> shared tokens/components -> bridge aliases -> module UI -> release evidence`

### Authority split

- Backend authority owns design config version/ETag, template registry, template lifecycle, impact reports, compliance matrix, debt/drift reports, rollout/apply/rollback, waivers, audit history and release blockers.
- `00b-theme-manager.js` remains the runtime token/theme engine. It may keep user preference and offline read cache, but it is not template registry authority.
- `00ba-graphics-governance-service.js` is the frontend service contract. UI code must call it instead of hardcoding endpoint actions in visual controls.
- LocalStorage is allowed only for preview cache, unsaved draft cache and last-view/filter/sort cache.

### Required backend actions

The frontend expects these action contracts to exist:

- template registry: `admin_template_registry_get`, `admin_template_get`, `admin_template_draft_save`, `admin_template_validate`, `admin_template_publish`, `admin_template_deprecate`, `admin_template_modules_get`
- impact analysis: `admin_graphics_impact_token`, `admin_graphics_impact_template`, `admin_graphics_impact_component`, `admin_graphics_impact_policy_pack`
- compliance/debt/drift: `admin_graphics_compliance_get`, `admin_graphics_non_compliant_get`, `admin_graphics_debt_get`, `admin_graphics_drift_get`
- rollout: `admin_graphics_rollout_stage`, `admin_graphics_rollout_apply`, `admin_graphics_rollout_rollback`, `admin_graphics_rollout_canary_apply`
- audit/waiver/blockers: `admin_graphics_audit_history_get`, `admin_graphics_waiver_create`, `admin_graphics_waiver_approve`, `admin_graphics_waiver_expire`, `admin_graphics_waiver_active_get`, `admin_graphics_release_blockers_get`
- leadership control-plane artifacts: `admin_graphics_change_set_get`, `admin_graphics_lineage_get`, `admin_graphics_runtime_beacon_get`, `admin_graphics_debt_observatory_get`, `admin_graphics_environment_policy_packs_get`, `admin_graphics_release_dashboard_get`, `admin_graphics_release_link_get`

Mutating actions must require optimistic concurrency through `expectedVersion` or `If-Match`.

### Module compliance states

Every module must be classified as one of:

- `full-admin-controlled`
- `bridged-to-shared-tokens`
- `legacy-private-css`
- `blocked`

Any non-compliant module must carry reason text, debt counts and remediation ownership. New modules or major refactors are blocked unless their graphics linkage is `full-admin-controlled` or an approved waiver covers the remaining bridged debt.

### Release evidence

Release manifest and evidence bundle must include graphics authority refs, template registry version/checksum, compliance matrix ref, impact analysis ref, waiver refs, drift report and rollback plan.

### Leadership control-plane artifacts

The backend projection must expose these machine-readable artifacts:

- `GraphicsChangeSet`: grouped graphics edit model with diff, impact, severity, rollout scope and evidence checklist.
- `ModuleGraphicsLineageGraph`: Admin -> backend -> tokens -> components -> template -> module route/screen lineage.
- `RuntimeGraphicsComplianceBeacon`: per-module runtime probe for shared token, hm-* component and private CSS linkage.
- `VisualDebtObservatory`: bridge alias, private CSS, hardcoded style and uncontrolled legacy shell debt by module/domain/team/route.
- `GraphicsEnvironmentPolicyPack`: office, review, admin, shopfloor, kiosk, TV and night-shift policy packs.
- `GraphicsReleaseDashboard`: blockers, waivers, drift/debt counts, rollout readiness and post-apply verification obligations.

Impact severity is fixed to `low`, `medium`, `high`, `regulated` and `shopfloor-critical`; evidence and QA rerun gates must be derived from that class.

## Ownership Model

Design authority owns:

- token taxonomy
- naming
- preview semantics
- governance rules

Platform core owns:

- runtime token mapping
- shared component contract
- loader/version integrity

Feature teams own:

- information architecture
- domain workflow
- bridge aliases for legacy migration

Feature teams do not own:

- private replacement of the platform graphics standard

## Bottom Line

The platform is no longer allowed to evolve by "best effort consistency".

The new rule is stricter:

- one graphics standard
- one runtime
- one admin control surface
- one review contract

Anything outside that path is technical debt on day one.
