ÁP DỤNG KHI
Dùng khi quản lý tài liệu QMS, mã nguồn cổng, định nghĩa biểu mẫu, manifest phát hành, biên nhận triển khai và hồ sơ đồng bộ ngược trên M365 SharePoint, nhưng vẫn phải biên tập cục bộ, quản trị bằng Git và triển khai lên máy chủ theo kiểm soát phát hành. ANNEX này khóa ranh giới giữa nguồn được kiểm soát, lịch sử phiên bản Git, bản đang chạy trên máy chủ và hồ sơ hoặc bằng chứng vận hành.

1. Mục đích, phạm vi và kết quả bắt buộc

ANNEX này quy định kiến trúc bắt buộc để HESEM đặt bộ nguồn trên M365 SharePoint mà không làm mờ chuỗi kiểm soát: máy cục bộ được phê duyệt dùng để biên tập, Git giữ lịch sử phiên bản, máy chủ chỉ nhận bản đã được phát hành, còn dữ liệu môi trường chạy phải tách hẳn khỏi thư viện nguồn. Người đọc phải xác định rõ bốn điểm: nguồn nằm ở đâu, máy nào được phép đồng bộ, Git giữ vai trò gì và dữ liệu nào tuyệt đối không được quay ngược vào cây nguồn.

KẾT QUẢ BẮT BUỘC: bộ tài liệu nguồn chỉ có một luồng chuẩn SharePoint local đồng bộ worktree → Git remote → máy chủ deploy; mọi hotfix buộc sửa trên máy chủ phải quay về Git → SharePoint local đồng bộ; dữ liệu môi trường chạy, bí mật hệ thống, binary nháp của luồng công việc, session Tệps, trạng thái rate-limit và dữ liệu giao dịch của biểu mẫu không được trộn vào QMS-Source-Kiểm soát.
Ranh giới tài liệu: ANNEX-133 khóa không gian, thư viện và đường dẫn; ANNEX-134 khóa cơ chế khởi tạo, phân quyền và tự động hóa; ANNEX-131 khóa từ điển field và list; ANNEX-135 khóa Tệp kế hoạch; WI-102 khóa thao tác tạo không gian, thư viện và quyền; WI-107 khóa thao tác đồng bộ cục bộ, Git và máy chủ.

4. Mô hình local đồng bộ bắt buộc

Đường dẫn local được phép

Chỉ các máy đã được IT/QMS chỉ định mới được đồng bộ thư viện QMS-Source-Kiểm soát. Worktree local phải nằm dưới thư mục đồng bộ do OneDrive quản lý trong hồ sơ người dùng, ví dụ:

%UserProTệp%\HESEM Engineering\HESEM-Digital - QMS-Source-Kiểm soát\01-Kiểm soátled-Source\eqms.hesemeng.com\

Git siêu dữ liệu được phép

Thư mục .git không được nằm trong thư mục đang đồng bộ với SharePoint. Git siêu dữ liệu phải được đặt ở đường dẫn tách rời ngoài vùng đồng bộ, ví dụ:

C:\HESEM-Git\eqms.hesemeng.com.git\

Lý do kỹ thuật bắt buộc

  • Microsoft cho phép đồng bộ thư viện SharePoint xuống máy qua OneDrive hoặc Tệp Explorer, nhưng không hỗ trợ đồng bộ vào network drive, mapped drive hoặc qua symbolic liên kết hay junction. Thiết kế local workspace phải bám đúng ranh giới này.
  • Git siêu dữ liệu thay đổi rất dày; đặt .git trong vùng đang đồng bộ sẽ làm tăng nguy cơ xung đột đồng bộ, Tệp lock và lịch sử nhiễu không liên quan đến nội dung thực.
  • Mô hình git work-tree tách khỏi git-dir cho phép worktree nằm trong vùng SharePoint đồng bộ mà không kéo toàn bộ Git internals vào thư viện đang đồng bộ.

6. Luồng promotion chuẩn và luồng reverse đồng bộ

Luồng chuẩn

Worktree local đang đồng bộ SharePoint chỉ được chỉnh trên admin endpoint đã đăng ký, sau đó commit và push lên Git remote, rồi máy chủ pull hoặc deploy. Sau khi triển khai xong, phải lưu deploy receipt tại 03-Máy chủ-Deploy-Receipts.

Luồng hotfix từ máy chủ

Chỉ dùng khi buộc phải sửa point-of-use trực tiếp trên máy chủ. Sau hotfix, thay đổi phải được commit và push về Git remote ngay; local admin phải kéo ngược về worktree đồng bộ; và hồ sơ reverse đồng bộ phải được đóng tại 04-Reverse-Đồng bộ-Intake.

Luồng biểu mẫu

Định nghĩa biểu mẫu, schema đã phát hành và source template đi theo luồng nguồn chuẩn. Biểu mẫu entry, phê duyệt, môi trường chạy archive và submission payload không được đi vào Git hoặc source thư viện trên SharePoint.

Cơ chế phát hành lên VPS thực tế (eqms.hesemeng.com — host chạy PHP 8.5 + nginx + PostgreSQL + Redis + RabbitMQ):
Đường mặc định: push vào branch main trên Git remote → GitHub Actions chạy .github/Luồng công việcs/deploy.yml → SSH vào VPS, gọi /var/www/eqms.hesemeng.com/dao cụs/vps-chuẩn bị máy/scripts/deploy.sh. Script này chụp ảnh dữ liệu môi trường chạy vào /var/www/data-private/.deploy-ảnh chụp trạng tháis/<TS>/, chạy reset có kiểm soát, sau đó phục hồi môi trường chạy theo xác thực sha256; nếu mismatch sẽ dừng và báo lỗi to.
Đường thủ công (khi CI không khả dụng): ssh eqms 'sudo -n bash /var/www/eqms.hesemeng.com/dao cụs/vps-chuẩn bị máy/scripts/deploy.sh' — bắt buộc dùng script này, không gõ trực tiếp git reset --hard, git restore, git làm sạch -fd trên VPS.
Hot-fix tại VPS (hiếm): chỉ chỉnh qua giao diện admin của cổng đi qua DataĐồng bộMutationService (ghi ảnh chụp trạng thái + đánh giá_events). Không sửa trực tiếp mom/data/config/*.json bằng vi/sed/cat>. Sau hotfix, đóng hồ sơ tại QMS-Source-Kiểm soát/04-Reverse-Đồng bộ-Intake/{YYYY}/ theo §6 luồng reverse đồng bộ.
Tuyệt đối cấm trên VPS: bất kỳ thao tác Git phá huỷ nào (git reset --hard, git restore, git làm sạch -fd) trên /var/www/eqms.hesemeng.com — đã có tiền lệ mất dữ liệu người dùng users.json khi rollback bằng git reset --hard origin/main qua SSH (sự cố commit 4a8ffebd); chính sách hiện hành neo tại CLAUDE.md §VPS deployment chính sách.

8.1. Luồng đồng bộ chi tiết: Onchuyền biểu mẫu → SharePoint

Khi người dùng gửi biểu mẫu trực tuyến trên web cổng, máy chủ phải xử lý tuần tự theo các bước sau:

BướcHành độngChi tiết kỹ thuật
1Người dùng gửi biểu mẫu trên web cổngMáy chủ nhận request POST tại api.php với payload đã qua kiểm tra phía client.
2Máy chủ lưu JSON entry làm bản gốcGhi vào qms-data/onchuyền-biểu mẫus/entries/{biểu mẫu_code}.json. Đây là bản gốc duy nhất trên máy chủ.
3Máy chủ tạo PDF ảnh chụp trạng tháiTên Tệp bắt buộc: FRM-{code}_V{ver}_{scope}_{YYYYMMDD}_{HHMM}-{UserID}.pdf.
4Gán Hồ sơ-ID khi loại hồ sơ yêu cầuÁp dụng cho NCR, CAPA, FAI và các loại hồ sơ cần mã định danh. Máy chủ phải cấp từ atomic counter sử dụng LOCK_EX để tránh trùng mã.
5Đồng bộ dữ liệu cấu trúc lên SharePoint ListMáy chủ gọi Microsoft Graph API để tạo hoặc cập nhật item trong SharePoint List tương ứng. Dữ liệu cấu trúc này phục vụ bảng điều khiển, báo cáo và truy vấn siêu dữ liệu.

9. Điều kiện không được go-live

  • Chưa tạo thư viện QMS-Source-Kiểm soát hoặc chưa khóa quyền đúng theo mô hình group.
  • Worktree local vẫn còn thư mục .git nằm trong vùng SharePoint đồng bộ.
  • Chưa chứng minh được trọn một vòng local đồng bộ → Git push → máy chủ pull → deploy receipt → local pull.
  • Môi trường chạy noise vẫn còn bị stage hoặc commit trong quá trình đồng bộ thử.
  • Chưa có quy tắc tách riêng giữa onchuyền biểu mẫu source và dữ liệu giao dịch của onchuyền biểu mẫu.
  • Chưa có WI thao tác cho admin endpoint và máy chủ người vận hành.
