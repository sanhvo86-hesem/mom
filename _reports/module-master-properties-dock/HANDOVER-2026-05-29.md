### Value resolver (v3-G12 / mm9)

```js
// Primary path — works for 95% of cases on this codebase
function resolveCssVarPx(cssVarChain){
  var rs = getComputedStyle(document.documentElement);
  for (var i = 0; i < cssVarChain.length; i++) {
    var raw = (rs.getPropertyValue(cssVarChain[i]) || '').trim();
    if (!raw || raw.indexOf('var(') === 0) continue;
    var n = parseFloat(raw);
    if (!isNaN(n)) return Math.round(n);
  }
  // Fallback: shadow-DOM probe (avoids body-level CSS clamping divs to 8px)
  try {
    var host = document.createElement('div');
    host.attachShadow({mode:'open'});
    document.body.appendChild(host);
    var inner = document.createElement('div');
    host.shadowRoot.appendChild(inner);
    var out = null;
    for (var j = 0; j < cssVarChain.length && out === null; j++) {
      inner.style.cssText = 'position:absolute;width:var(' + cssVarChain[j] + ',99999px);height:1px';
      var w = parseFloat(getComputedStyle(inner).width);
      if (!isNaN(w) && w >= 0 && w < 99999) out = Math.round(w);
    }
    document.body.removeChild(host);
    return out;
  } catch (e) { return null; }
}
// resolveCssVarColor() is the parallel for hex/rgb values
```

## 6. Known gotchas

1. **`//` comments inside string-concat chains parse as `+` unary operators → NaN.** Always use `/* */` block comments inside JS concatenation:
   ```js
   var html = ''
     + '<div>...</div>'
     // <-- THIS COMMENT BREAKS THE CHAIN, INSERTS NaN
     + '<div>...</div>';
   ```
   Fixed in v3-G11 commit `0cd0822ab`. Pattern caught users' "NaN" stray text bug.

2. **CSS `calc(-100% - 32px)` in `transform: translateX()` silently doesn't apply** in some Chrome versions. Use plain percentage (`translateX(-110%)`) with `!important` to win specificity.

3. **`getPropertyValue('--o3-space-md')` returns the literal `"var(--o3-space)"` string** when the variable is undefined. With orders-v3.css loaded it returns the resolved `"8px"`. The resolver handles both — but if you see literal `var(...)` text, it means orders-v3.css isn't loaded on the VPS.

4. **Body-level probe divs get clamped to 8px width.** Some CSS rule on `*` or `body div` sets a min-width that overrides inline `width: var(--x)` to 8px (the master space value). The shadow-DOM probe fallback in `resolveCssVarPx` works around this.

5. **Inline overrides on `:root` persist across reloads.** When you set `document.documentElement.style.setProperty('--o3-space-md', '6px')` from an editor change, that inline style persists until cleared. The dock displays the CURRENT computed value, so previous user edits appear as "current". To reset: `document.documentElement.style.removeProperty('--o3-space-md')`. Consider adding a "Reset to default" button in v3-G14.

6. **Sidebar covers dock at left:0.** The 260px sidebar visually covers `left:0` even with z-index 9000. Hence the `left:260px` offset. If sidebar gets a collapse mode in future, watch out for body class triggers — already handled for `body.sidebar-collapsed,is-sidebar-collapsed,[data-sidebar=collapsed]` → `left:60px`.

7. **Per-input dedupe via `data-mod-sample-wired` attribute.** Without this, every section-switch double-attaches the input listener and edits fire 2N times. The `wireInlineTokenEditors` function checks `:not([data-mod-sample-wired])` selector. When re-rendering the dock body, the renderer strips wired flags from doomed inputs first (`_renderAdmModuleSampleHtml` start).

   - User checks 3 boxes, changes 3 values, clicks "Lưu cho tổ chức"
   - POST hits `/api/v1/graphics/overrides` with 3 rows
   - Refresh page → 3 boxes still checked with saved values, other 100+ rows still unchecked
7. **Dark mode + override**:
   - Switch to dark mode → all colors flip to dark palette
   - Custom-override on a color row uses the user's chosen color regardless of mode (or have per-mode overrides — design decision)
8. **Reset to defaults** button on Global Theme → all overrides cleared, returns to research-backed values
