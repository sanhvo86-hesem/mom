## 1. Purpose and Governance Posture

The Analytics & AI domain computes operational intelligence from HESEM's event and data streams and exposes advisory features to domain users. The governance posture for all AI features is categorical: AI is advisory only. No AI feature in HESEM will make, emit, transmit, or take effect as an authoritative action on any of the 36 banned decision types (BD-1..BD-36). The banned-decision triple-defense (CI linter + L2 runtime middleware + OTG axiom A-05) enforces this invariant across the architecture; this domain owns the application-layer expression of that constraint through the Banned-Decision Attempt Log resource.

Every AI feature is classified on the NIST AI RMF 1.0 risk tier scale (Tier-1 low risk through Tier-4 high risk). No HESEM feature currently exceeds Tier-3 (substantial risk; closest is the Complaint NLP Classification which touches patient safety signals). EU AI Act classification is maintained per each feature's Red-Team Report.

The domain provides: operational analytics data products, AI advisory invocations, model lifecycle management, governance records (decision records, override records, drift events), and the AI governance ledger that satisfies 21 CFR Part 11, EU GMP Annex 11, and EU AI Act accountability requirements.

## 2. Resource Families

### RF-C13-01 — KPI Snapshot

A KPI Snapshot is an immutable point-in-time computation of a Key Performance Indicator for a tenant, capturing the metric value, the grain (per-machine, per-shift, per-site), the data lineage pointer (source events), and the computation timestamp.

**Fields:** snapshot_id, tenant_id, kpi_code (ISO 22400-2 aligned; e.g., OEE-01, FPY-07, COPQ-03), grain (EQUIPMENT|SHIFT|SITE|TENANT), period_start, period_end, value (numeric), unit, confidence_score (0.0–1.0; reflects data completeness), source_event_range (first/last otg_event_id consumed), computation_model_version, pack_id (optional).

**Lifecycle:** KPI Snapshots are immutable once created (append-only table). Corrections are expressed as a new snapshot with supersedes_snapshot_id pointer. The superseded snapshot retains its immutable record for audit trail.

**State machine:** KPI Snapshots do not have a mutable lifecycle; they are write-once. The computation job itself transitions through: SCHEDULED → RUNNING → COMPLETED | FAILED. If FAILED, the job logs the failure to the Drift Event resource (RF-C13-08) and retries on next interval.

**RACI:** R=Data Platform Lead, A=AI Lead, C=Domain leads (as consumers), I=Tenant Admin.

**Failure modes:**
- FM-C13-01-A: Source events missing for computation period (CDC lag > SLO-13 < 60s threshold) → snapshot created with confidence_score < 1.0 and `data_incomplete` flag; consumers display "(Partial Data)" indicator.
- FM-C13-01-B: Computation job FAILED three consecutive intervals → Drift Event raised; SRE alert; manual review of pipeline health.

**KPIs:** KPI computation job success rate ≥ 99.5% per 7-day window; no snapshot with confidence_score < 0.8 published without advisory warning to consumers.

### RF-C13-02 — Score

A Score is a computed rank or probability value associated with a specific HESEM root record, generated by an ML model. Examples: predictive maintenance failure probability, NC complexity score, supplier quality risk score.

**Fields:** score_id, tenant_id, root_kind, root_id (hashed for PII), model_version_id, score_type (FAILURE_PROBABILITY|RISK_RANK|COMPLEXITY|QUALITY_RISK), value (float), confidence_interval_lower, confidence_interval_upper, explanation_tokens[] (LIME/SHAP token list for explainability per NIST AI 100-1 §2.5), computed_at, expiry_at (score is stale after; UI suppresses display).

**Lifecycle:** Scores are immutable once computed. Expiry is model-specific (predictive maintenance scores expire after 24 hours; supplier quality scores expire after 7 days). After expiry, the score is archived and a fresh computation is triggered on next inference cycle.

**RACI:** R=AI Lead, A=Data Platform Lead, C=Domain leads (as consumers), I=Compliance Lead (for regulated-domain scores).

**KPIs:** Score freshness: 100% of active root records have a non-expired score for enabled score types; p95 score computation latency < 500ms for online inference; explanation token coverage ≥ 90% of top-3 contributing features.

## 4. Banned-Decision Routing (per L1 §3)

The banned-decision intercept operates at three layers (triple-defense, per B6 C2 and B4 §4):

**Layer 1 — CI Linter:** Static analysis during model training pipeline checks that the model's decision taxonomy does not include BD-coded outputs. Models trained to output BD-adjacent decisions are blocked from reaching STAGING.

**Layer 2 — Runtime Middleware (L2):** Every Advisory invocation passes through the L2 authority middleware before the output is delivered to the UI. The middleware pattern-matches the output against the BD-1..BD-36 catalog. On match: output is stripped; `banned_flag=true` is set; the advisory message is replaced with the human-authority notice; a Banned-Decision Attempt Log entry (RF-C13-07) is created; `hesem_banned_decision_attempts_total{layer=runtime, bd_id=BD-N}` counter incremented → SEV-1 immediate alert.

**Layer 3 — OTG Axiom A-05 (AI_NOT_BANNED_PRINCIPAL):** The OTG axiom check verifies that the `principalid` in any `otg_event` with a BD-sensitive predicate is a human principal, not an AI service account. Any AI service principal appearing in such a position triggers OTG axiom violation → SEV-1.

The routing ensures that regardless of model output, banned decisions cannot reach any surface that would allow them to be applied. The application-layer expression is that advisory features with BD proximity (F-02, F-04, F-06, F-13, F-14, F-19, F-31) have explicit notice: "This advisory requires qualified human review and authority action."

## 6. Failure Modes (Including Red-Team Finding Integration)

**FM-C13-01 — Model produces BD-adjacent output in production:**
Severity SEV-1. Triple-defense activated. Banned-Decision Attempt Log created. Model moved to DEPRECATED immediately; investigation opened. Red-team finding logged. No further invocations until root cause resolved and new model version passes red-team.

**FM-C13-02 — Inference sub-processor outage:**
Advisory invocations time out at 30 seconds → ABSTAINED state. UI shows abstain message. Human proceeds without advisory. Sub-Processor Security Event (RF-C13-10) created if outage exceeds 30 minutes. Advisory is non-blocking by design; workflow continues.

**FM-C13-03 — Training data PII exposure:**
If corpus reference (RF-C13-05) is found to contain un-pseudonymized PII, model version is immediately moved to DEPRECATED; corpus reference flagged; DPO notified; DPIA re-evaluated; affected users notified per GDPR Art 33/34 timeline.

**FM-C13-04 — Override rate exceeds drift threshold:**
Drift Event raised (RF-C13-08) with severity MINOR (10% increase) or MAJOR (25% increase) relative to 30-day baseline. MAJOR immediately triggers model DEPRECATED state + routing to fallback model or abstain-all mode.

**FM-C13-05 — Red-team reveals prompt injection vulnerability (LLM features):**
Feature immediately disabled (feature_enabled_for_tenant = false for all tenants). Red-Team Report created with FAIL status. Model returned to EXPERIMENTAL. Fix assessed per OWASP LLM01 (Prompt Injection) remediation guidelines. No re-deployment until OWASP checklist cleared and new red-team passes.

**FM-C13-06 — Explainability tokens unavailable:**
Score resource created without explanation_tokens[]. EU AI Act Art 86 (right of explanation for high-risk AI) may be compromised for Tier-3 features. Alert AI Lead. Feature-level KPI `explanation_coverage` drops below threshold → Drift Event.

**FM-C13-07 — AI Decision Record write failure:**
Advisory delivery is blocked until the Decision Record is written successfully (same transaction). If the transaction fails, the advisory is not delivered (fail-safe). Error logged; SRE alert; investigation of persistence layer health.
