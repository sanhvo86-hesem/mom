# 07 — Instructions for writing QMS document content

> Version: V0 | Effective: 2025-06-01 | Owner: QMS Engineer

---

## 1. Writing principles

### 1.1 Fundamental principles

| # | Principle | Explain |
|---|-----------|-----------|
| 1 | Short, clear sentences, used for execution | Each sentence has a maximum of 25 words. The reader must immediately understand what to do. |
| 2 | Subject + action + object | Always clearly state who does, WHAT, and with WHAT. |
| 3 | Do not use vague explanations | Do not write "to ensure", "contribute to improvement", "towards goals". |
| 4 | No meta-text | Do not write "this document is intended to...", "this section describes...". |
| 5 | No "AI generated" | Do not write "generated by AI", "AI generated", "auto-generated". |
| 6 | Every sentence must serve a real operation | Delete any sentences that don't lead to a specific action or decision. |
| 7 | One idea, one sentence | Do not cram multiple ideas into one sentence with consecutive commas. |
| 8 | List format is preferred | When there are 3 or more sections, use bullets or tables instead of paragraphs. |
| 9 | Do not include editorial notes in the body of the SOP | Do not insert notes like "addition to note", "link note", "rules for using terminology", "different from previous version" into operational documents. |
| 10 | Do not state “new from previous version” in unreleased SOPs | When the document is still a first pre-release draft, keep `V0` and do not describe differences from the old draft in the body. |
| 11 | Do not insert the rationale benchmark into the body of the SOP | Reasons for choosing KPIs, source benchmarks and inference notes to put in working notes / core-standard / drafting records, not included in the SOP body. |
| 12 | Format standardization is allowed at the core, content must be upgraded according to each SOP | Palette, layout, table structure, technical checklist can be locked in core-standard; But the content of Section 1-8 must be researched and written according to each document, not copied in bulk. |
| 13 | Do not use hybrid languages ​​for role/department boundaries | Do not write `Customer Dịch vụ`, `Process Owner`, `all functions`, `Quy trình Owner`, `Department Head`. |
| 14 | Don't cover the coverage gap with generic phrases | If departmental scope is missing or JD is missing for a recurring decision, the gap must be noted and the handbook/core-standard/JD updated according to the rule. |
| 15 | The organizational handbook/matrix must not keep the placeholder ambiguous | In the handbook or function level interface table, do not put `các phòng liên quan`, `các phòng chức năng`, `line manager`, `line owner`, `bộ phận chuyên môn`; must resolve to `D-code` or `role code`. |
| 16 | Do not keep English in the SOP body other than the locked exception list | Only standard abbreviations, system proper names, `Setup`, `Traveler`, `Balloon`, `Kaizen`, `Dreyfus`, `Pareto`, `Kolb`, `Poka-yoke` and technical proper nouns locked at `03-language-and-translation.md` are kept. |

### 1.2 Tone of writing

- The tone is commanding, professional, not flippant.
- Do not use "you", "we", "us".
- Use specific role names: "QMS Engineer", "QC Inspector", "Production Planner".
- When speaking in general, use "executor" or "assignee".
- For documents that have not been officially released for the first time, the version header is always `V0`.

## 2. Command structure

### 2.1 Correct structure

Each statement in SOP/WI follows the pattern:

```
[Vai trò] + [hành động] + [đối tượng] + [điều kiện/thời điểm nếu có].
```

### 2.2 Example of true vs false

| Type | Sentence |
|------|-----|
| CORRECT | QMS Engineer reviews document structure before sending for approval. |
| WRONG | This document is designed to assist in reviewing the structure... |
| CORRECT | Do not release documents without cross-review. |
| WRONG | It is advisable to ensure that the document is cross-reviewed before release. |
| CORRECT | QC Inspector measures 5 points according to the drawing before accepting the batch. |
| WRONG | Measurements need to be done carefully to ensure quality. |
| CORRECT | Shift Leader stops the machine as soon as it detects 3 consecutive defective parts. |
| WRONG | In case a problem is detected, consider stopping the machine for troubleshooting. |

### 2.3 Rules for writing negative commands

- Use "NO" + verb: "Do not release without a signature."
- Do not use "Should not", "Should be avoided" — too weak, not binding.

### 2.4 Rules for writing conditions

- Use "When" or "If" at the beginning of a sentence for conditions.
- "When NCR is open for more than 5 days, QA Manager reports to CEO."
- Don't use "In case", "For the situation" — long and vague.

## 3. Standard vocabulary — Required level

### 3.1 Three levels

| Keywords | English | Meaning | Use |
|---------|-----------|---------|-----------|
| **RIGHT** | shall | Mandatory implementation. Violation = non-conformance. | Used for ISO, customer, legal requirements. |
| **SHOULD** | should. should | Strong recommendation. Exceptions are allowed for reasons. | Used for best practice, internal recommendations. |
| **MAYBE** | sewing | Options. The implementer decides for himself. | Used for alternative and complementary methods. |

### 3.2 How to mark up in HTML

```html
<span class="req-tag shall">PHẢI</span>
<span class="req-tag should">NÊN</span>
<span class="req-tag may">CÓ THỂ</span>
```

### 3.3 Rules of use

- Every SOP MUST have at least 1 "MUST" requirement in each gate.
- DO NOT use "MUST" for a recommendation level — it will cause non-conformance when auditing.
- When unsure of the level, default to "SHOULD" and let QA Manager confirm the upgrade.

## 4. SOP content rules — 10 sections

### Section 1: Purpose

- Allow `1 câu điều hành rất ngắn` to start if the sentence correctly locks in the actual workings of the SOP.
- After the opening sentence, use `3-5 bullet points`.
- Every bullet begins with **verb** (Establish, Control, Ensure, Regulate, Prevent).
- DO NOT begin with a noun ("This procedure...", "This document...").
- The opening sentence cannot be more than 1 sentence long, cannot be meta-text, and cannot substitute for executive bullets.
- Must be inferred from the risk of being blocked, the decision being held and the output being locked in Section 6 / 7 of the SOP itself.
- Do not reuse the same set of bullets for multiple SOPs just by changing the nouns.

**For example:**
```
- Thiết lập quy trình kiểm soát tài liệu từ tạo mới đến phê duyệt.
- Đảm bảo mọi tài liệu phát hành đều có rà soát chéo.
- Ngăn ngừa sử dụng tài liệu hết hiệu lực tại nơi làm việc.
```

### Section 2: Scope

Divided into 2 clear parts:

**Covered:**
- List objects, processes, and areas of application.

**No substitution:**
- List the documents/processes that are NOT within the scope of this SOP.
- Boundary must adhere to the actual upstream / downstream handoff in Section 7, not using decorative document lists.

**For example:**
```
Có bao phủ:
- Tất cả tài liệu QMS nội bộ: SOP, WI, ANNEX, FRM, JD.
- Tài liệu gốc từ khách hàng khi được nội bộ hóa.

Không thay thế:
- SOP-104 (bảo mật dữ liệu và IP).
- Quy trình kiểm soát bản vẽ khách hàng do Engineering quản lý.
```

### Section 3: Terminology & principles

2 column table:

| Terminology | Terms of use |
|-----------|-----------------|
| Traceability (traceability) | Used to refer to the ability to trace forward/backwards with evidence |
| System of Record - SoR (standard recording system) | Used for native transaction systems such as ERP |
| Single Source of Truth - SSOT (single standard source) | Used for storing controlled standard records |

- List only terms used in this SOP.
- DO NOT copy the entire system glossary into each SOP.
- The term name column MUST follow the pattern `English term (thuật ngữ tiếng Việt chuẩn)`.
- In the document body, priority is given to using the Vietnamese version listed in Section 3.
- Terminology is retained only when absolutely necessary to understand the gate, step, hold, release or exception of that SOP.
- DO NOT write half English and half Vietnamese like `mixed source kiểm soát`, `job close tài chính`, `trace gap`, `first-piece release` if the SOP already has a standard Vietnamese version.
- Only keep standard abbreviations or codes when this is a common operating language and does not obscure the meaning, for example `FAI`, `SPC`, `Cpk`, `NCR`.
- If the author adds a new glossary term, he must enter `meaning` as the full English name before writing `vi` and `def`; Do not create canonical rows of the form `Full Term (ABBR)`.
- Exceptions that have been hardwired for the entire system are: all standard abbreviations; `Setup`, `Traveler`, `Balloon`; `Kaizen`, `Dreyfus`, `Pareto`, `Kolb`, `Poka-yoke`; and proper noun system/brand in `03-language-and-translation.md`.
- In addition to the above exceptions, the text displayed in the SOP must be in Vietnamese and not left in English just because of internal use.
- Do not add notes explaining the rules for writing terminology right in the body of the SOP; rule belongs to `core-standard`, not to operational documentation.

### Section 4: Roles, authorities & RACI

3-column table:

| Role | Responsibility | Permissions/Block Points |
|---------|------------|------------------|
| QMS Engineer | Review structure, check cross-ref | Right to refuse release if cross-review is lacking |
| QA Manager | Approve SOP level documents | Block point: No approval = no release |
| QMS[DC] or appropriate JD-linked role bundle | Draft, update, and monitor effectiveness according to SOP scope | Responsible for content according to the assigned role boundary |

- The "Permissions / Blocking" column is important — it shows who has the authority to stop the process.
- Every owner holding the gate in Section 6 and every role with handover/removal authority in Section 7 must appear in Section 4.
- If the SOP uses a RACI matrix instead of a 3-column table, it MUST still clearly state who has the right to block and who has the right to unhold.
- Do not use ambiguous placeholders such as `Process Owner`, `Department Head`, `Responsible Person`, `Data Owner`, `Top Management` in Section 4 role lines; must resolve to JD-linked role code or explicit role bundle.
- Header, Section 4, Section 6 and Section 8 prioritize the chip code/JD link role; The body of the new paragraph uses the full English name when necessary.

### Section 5: Inputs, outputs & prerequisites

4 field boxes:

| Box | Content |
|-----|---------|
| **Input** | What documents are needed before starting this process |
| **Output** | What product/record is created upon completion |
| **Prerequisites** | Which conditions MUST be satisfied before starting? |
| **Trigger** | What event triggers this process? |

- The input must be mapable to the state before `IG1 / B1`.
- The output must be mapable to the state after the last gate or last step.
- The trigger must reflect the actual activation, restart, transfer, change or escalation event in the SOP.
- Do not write generic boxes of the type `khi cần`, `theo yêu cầu`, `tài liệu liên quan`.

### Section 6: Control gates, mandatory stops & KPIs

**Internal Gates (IG):**

Section 6 is not the place to recount the entire process. This section only describes:
- which ports must be kept/open,
- Who has the right to open the gate?
- measured HOLD condition,
- Minimum KPI or profile.

**Writing rules:**
- Use **IG table**, do not use gate cards.
- Not fixed 5 IG.
- Do not force the IG number to equal the number of steps detailed in Section 7.
- Only create IG when there is a real `HOLD / RELEASE` score.
- KPIs must be tied to the control effectiveness of each gate, not decorative KPIs.
- Each KPI MUST have at least a numerical threshold or SLA, a standard data source, and a trigger that responds to deviations.
- When using external benchmarks, it must be clear whether this is a reference benchmark or an internal target; Do not copy external numbers into the SOP without explaining the operating logic.
- Before rewriting Section 6, MUST read the old documentation and finalize the control architecture according to `13-sop-research-redraft-method.md`.
- The KPI sentence should be read according to the structure: `metric + ngưỡng + nguồn dữ liệu hoặc operating context + trigger phản ứng`.
- Each KPI should prioritize sticking to one of the groups: `đúng ngay lần đầu`, `tốc độ phản ứng`, `độ đầy đủ bằng chứng`, `ổn định quá trình`, `đóng action đúng hạn`, `độ chính xác dữ liệu/chứng từ`, `zero-escape / zero-orphan / zero-unauthorized`.
- Sample KPIs by common family:
  - Contract review / ACK: `<= 1 ngày làm việc`, `mismatch sau commit = 0`.
  - Planning / dispatch: `schedule attainment >= 90%`.
  - First-piece / FAI: `pass ngay lần đầu >= 95%`.
  - Final release / documents: `document accuracy >= 99.5%`, `thiếu chứng từ bắt buộc = 0`.
  - Access control: `cấp/đổi/thu hồi quyền <= 1 ngày làm việc`, `orphan account = 0`.
  - Backup / restore: `backup success >= 99%`, `restore test = 100% theo chu kỳ`.
  - MSA / capability: `GRR < 10%` is good, `10–30%` is only used conditionally; `Cpk/Ppk >= 1.33` is a common reference level.
- The above numbers are the starting point for designing actual KPIs; It is allowed to tighten more according to risk, but not to loosen unreasonably.

**KPI metrics:**

| KPI | Target | Measurement | Frequency |
|-----|---------|---------|---------|
| Document approval time | <= 5 working days | Date from submit to approval | Monthly |
| Percentage of expired documents still in circulation | 0% | Number of expired documents / total | Every quarter |

KPI is not up to standard if only written as:
- `đúng hạn`, `được kiểm soát`, `đủ hồ sơ`, `cải thiện liên tục`

Qualified KPIs should be able to answer at the same time:
- What is the threshold?
- Measure from any system or profile.
- Regardless of the deviation, hold the gate, escalate or open the action.

### Section 7: Detailed process

Section 7 is the **execution workflow**. Each detailed step should be separated according to:
- change roles,
- change areas/processes/systems,
- change main resource (material / machine / tool / fixture / gage / program),
- quality control point,
- handover or revalidation point.

**Writing rules:**
- There is a flowchart at the beginning of Section 7.
- The number of bubble flowcharts must match the number of detailed step headings.
- Do not force the number of steps according to the IG number.
- With a real operational SOP, 8–14 steps is normal if the process has many handoffs.
- Only separate steps when there is a change in roles, key resources, system states, checkpoints, revalidation, containment or actual handover.
- Before rewriting Section 7, MUST read the old document and research the official external source for that SOP.
- Bubble flowchart must use a rotating color palette and the same color logic as the `proc-num` of the detail headings.
- With new SOPs generated by script, each `flow-step`, `flow-num` and `flow-arrow` should have an inline style according to the palette; Global CSS is just a fallback layer, not the only layer.
- Classes `.active` and `.critical` are just add-on classes for decision/test semantics; Don't break the numbering and palette logic with each step.
- Before considering complete, `flowchart steps = proc-num headings = số bước thực thi` must be checked.

**For example:**
```html
<h3><span class="proc-num">3</span> Rà soát chéo</h3>
<p>Người được chỉ định rà soát chéo kiểm tra nội dung và cross-reference.</p>
<ul>
  <li>Đọc toàn bộ tài liệu, đối chiếu với SOP/WI liên quan.</li>
  <li>Kiểm tra mọi liên kết nội bộ còn hoạt động.</li>
  <li>Ghi nhận kết quả vào FRM-105 Peer Review Log.</li>
</ul>
<div class="note-soft">Thời gian rà soát chéo: tối đa 3 ngày làm việc.</div>
<div class="role-note">Vai trò: Peer Reviewer (do Document Owner chỉ định)</div>
```

### Section 8: Exceptions, changes & redo

- By default, a 5-column exception table is used.
- Bullet lists are only allowed when the SOP is truly a narrow governance document, the number of scenarios is small, there are not many different owners/holds, and does not take away the ability to audit decisions.
- Each line must read: `tình huống + quy tắc xử lý bắt buộc + chủ trì + người gỡ hold hoặc phê duyệt tiếp + hồ sơ`.
- The exception situation must be inferred from the actual hold, restart, revalidation, partial release, waiver, system-down or change path point in Section 6 / 7.
- Do not add decorative exceptions that do not result from the actual operation of the SOP.

**For example:**
```
- Khi tài liệu khẩn cấp (an toàn, pháp luật): CEO có thể phê duyệt
  ngay, bỏ qua bước rà soát chéo. Ghi lý do vào DCR.
- Khi người rà soát chéo vắng mặt quá 3 ngày: Document Owner đề xuất
  người thay thế, QA Manager chấp thuận.
```

## 5. Special rules according to document type

### 5.1 WI (Work Instruction)

- Write in **step-by-step** format (step 1, step 2...).
- All stories begin with the verb menh lenh.
- Every meeting costs 1 dong. DO NOT combine multiple actions into 1 session.
- There are pictures/diagrams if the problem is complicated.
- Ends with: records can record + examiner.

### 5.2 ANNEX

- Write in **rule-pack** format: state, condition, rule.
- DO NOT write wall-to-wall.
- Every table MUST have a clear header and source notes.
- Use `iso-map` to link the ISO clause.

### 5.3 JD (Job Description)

- Responsibility for writing is complete, not paragraph writing.
- Every channel has: content + frequency + related records.
- The authority part (auth-grid) clearly states: what decisions can be made and to what extent.

### 5.4 Forms (Excel)

- Only have the name, short label, dropdown.
- DO NOT write long explanations in the form.
- Male explanation in SOP/WI is correct, NOT male in form.

## 6. Check before releasing

Before sending documents for approval, check:

| # | Category | Check |
|---|---------|---------|
| 1 | Every sentence has a clear subject | There are no passive sentences where it's unclear who did it |
| 2 | Every gate has a PASS/FAIL condition | There are no ambiguous gates |
| 3 | Every role in Section 4 appears in Section 7 | There is no "ghost" role |
| 4 | Any linked form/WI exists | No dead links |
| 5 | No meta-text | Delete "This document has no purpose..." |
| 6 | There is no "AI", "generated" | Check your accounting |
| 7 | Keywords MUST/SHOULD/CAN be used at the correct level | QA Manager confirmed |
| 8 | Glossary Elephant Consistency Terminology | Do not use 2 articles for 1 declaration |
| 9 | A4 printable | Print and check layout |
| 10 | Cross-reference correctly | All documents and links work |
