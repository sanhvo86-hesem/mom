# Domain Command Specification

This file defines the mandatory command layer that must replace direct business mutation through Generic CRUD or fragmented JSON services. Every command below is a backend API contract, not frontend behavior. Frontend must call process commands only after these APIs exist and are marked runtime-safe.

## Common Command Contract

All commands must implement the same envelope:

```http
POST /api/v1/commands/{CommandName}
Idempotency-Key: <required-unique-key>
X-Correlation-Id: <required-correlation-id>
X-CSRF-Token: <required-for-browser-session>
```

Common rules:

- The command starts one PostgreSQL transaction and commits only after all state, ledger, audit, evidence, and outbox writes are durable.
- Idempotency is checked before domain work. Same key and same fingerprint returns the same response. Same key and different fingerprint returns `409 idempotency_conflict`.
- All governed commands call `CreateElectronicSignature` or validate an existing signature when the command is regulated, financial, shipment-release, quality-disposition, or override-related.
- All commands write `command_audit`, domain-specific audit trail, and outbox event rows.
- Every error response includes `error.code`, `error.message`, `correlation_id`, `command`, `retryable`, and `violations[]`.
- Generic CRUD may not call these command internals.

## Compatibility Guards Already Applied

These runtime guards exist only to stop known P0 bypasses while the command layer is being built:

| Guard | Current behavior | Required command replacement |
| --- | --- | --- |
| Governed Generic CRUD mutation | `GenericCrudController` rejects governed create/update/delete/transition with `409 domain_command_required`; internal backfill requires env+header override. | All commands in this spec. |
| Quote conversion retry safety | Legacy quote conversion now uses idempotency plus a conversion lock and repairs stale quote state by returning the already-created SO. | `ConvertQuoteToSalesOrder` in one PostgreSQL transaction with unique `source_quote_id`. |
| Quote/SO/logistics document counters | Legacy quote/SO and logistics counters now use file locks around read-increment-write. | PostgreSQL sequence or `record_counters` row lock used by all command APIs. |
| Idempotency replay authority | PostgreSQL idempotency ledger now uses `scope_key_hash` for long scopes and refuses to reclaim `in_progress` rows even after TTL expiry; file/cache fallback also fail-closed on `in_progress`. | Command framework owns idempotency and exposes stale in-progress recovery only through an operator-controlled reconciliation command. |
| Engineering readiness before JO | Legacy SO transition to `engineering_ready` requires release package fields, and legacy JO creation rejects confirmed-only SOs. | `ReleaseSalesOrderToProduction` and `CreateJobOrder` with released master-data version locks. |
| Engineering readiness status | Missing `engineering_release_status` no longer defaults to released; a released/approved/complete status is required. | Released engineering package table is locked and versioned. |
| OQC fail containment | Legacy `oqc_update(result=fail)` creates/reuses an OQC-sourced JSONL NCR and active SO hold, and ignores client-supplied fake `ncr_reference` for failed results. | `RecordOqcResult` plus `CreateNcrFromQualityFailure` in one transaction. |
| Shipment gate on logistics routes | Legacy `packing_update(status=shipped)` and `delivery_confirm` call `ShipmentGateService`. | `ConfirmPacking` and `ConfirmDelivery`. |
| Shipment gate config normalization | `ShipmentGateService` accepts `gate_items` by normalizing it to `gates`. | Generated shipment gate policy from canonical quality/order authority. |
| Finance closed-period memo posting | Legacy AP/AR debit/credit memo creation rejects closed periods unless a matching approved backdate exception exists and is consumed under the finance-control lock. | All posting commands call one period policy before ledger/AP/AR/GL/COPQ/payment writes. |
| Supplier scorecard formula | Legacy scorecard now computes PPM, OTD, SCAR severity/open/overdue including still-open prior-period SCARs, supplier audit, and ASL/cert risk from supplier-quality JSON stores. | Canonical scorecard projection from PO receipt, IQC, SCAR, audit/cert, COPQ, OTD, and payment/release gates. |

None of these guards changes the acceptance rule: a domain is not runtime-complete until its command has PostgreSQL transaction authority, idempotency, audit/evidence, outbox, and tests.

## Compatibility And Rollback

- During migration, legacy JSON APIs may remain read-only behind feature flag `legacy_runtime_read_enabled`.
- Command services must dual-write JSON only in `SHADOW_WRITE` mode until reconciliation passes.
- Rollback from `POSTGRES_PRIMARY` to JSON is allowed only if drift report has zero missing rows for governed collections and no command was accepted after the rollback snapshot time.
- Existing endpoints remain but return `410 gone` or `403 forbidden` for governed mutations once command API is live.
