# M365 Folder Blueprint — Canonical per ANNEX-133 + ANNEX-139

This directory materializes the M365 folder structure that HESEM SharePoint
tenant should be provisioned with. Every directory below this `README.md`
traces letter-for-letter to a section of canonical:

- **Canonical sources**:
  - ANNEX-133 — site/library/dept/job-dossier topology (§2-§12)
  - ANNEX-139 — internal L2/L3 for 6 IP/restricted libraries: Part-REV-Master, Customer-Received, Tooling-Fixture-Gage, HR-Operations, Cleanroom-Records, Subcontractor-Records (§2-§8)
  - ANNEX-141 — semi-equipment-parts CNC supplier compliance (Cleanliness Pack 6, 4-OEM matrix, special-process map)
  - ANNEX-142 — M365 permission model (Entra ID + PIM + Conditional Access + Purview labels + Information Barriers)
  - **ANNEX-143 (v3, 2026-05-16) — Department × Role × Scenario × File-Class master matrix.** Drives §8 Operations subfolder expansion from generic "01-Quality-Planning" leaves to scenario-driven sub-folders (e.g. 20 scenarios for DEP-QA, 25 for DEP-SCM, 22 for DEP-FIN/HR, 25 for DEP-EHS). Adds 4 new departments: DEP-METRO, DEP-SP, DEP-MNT, DEP-LEGAL.
- **Builder script**: `tools/m365-provisioning/build_blueprint.py` (idempotent — re-running rebuilds from spec; preserves top-level user docs README.md + manifest.json)
- **Tree manifest**: `_TREE.txt` (sorted list of every directory, regenerated by the builder for audit-diff)

## Compliance posture

Strict letter-for-letter compliance with ANNEX-133 §2-§12.
**No invented folders. No invented libraries. No tokens outside §12.**

Any divergence from ANNEX-133 is documented in the "Known canonical
inconsistencies" section below, NOT silently corrected — the blueprint
keeps the canonical literal text so source-doc upgrades remain the
single point of truth.

- `RoleCode` (vd `ROLE-QAM`, `ROLE-SET`) dùng cho **tên THƯ MỤC** —

  hoặc người ký. Xuất hiện trong file pattern

## Known canonical inconsistencies (giữ nguyên — chờ upgrade ANNEX-133)

Trong khi xây blueprint phát hiện một số điểm canonical doc tự mâu thuẫn.
Blueprint **giữ nguyên literal text** từ canonical (strict compliance) và
liệt kê ở đây để source-doc author cân nhắc khi nâng cấp ANNEX-133.

1. **`DEP-PRO` vs `D-PROD`** — §8 line 383 path column ghi
   `Department-Ops/DEP-PRO/` (3 chữ); cùng dòng cell mã phòng ban
   trong §2 và §12 token table dùng `D-PROD` (4 chữ). Blueprint dùng
   `DEP-PRO` theo §8 vì đó là section enumerate path chính thức.

2. **Training-Records folder name có diacritic + space** — §6 lines 278-279
   literal:
   - `Training-Records/01-Đào tạo-kế hoạch/{YYYY}/`
   - `Training-Records/02-Điểm danh-and-Class-Records/{YYYY}/`

   Mâu thuẫn với §12 line 547 `{DescriptionSlug}` rule "Slug ngắn,
   không dấu tiếng Việt, không có khoảng trắng, dùng dấu gạch nối"
   và với khả năng SharePoint/OneDrive sync (dấu tiếng Việt và space
   gây encoding issues). Blueprint **giữ nguyên** literal canonical
   text. Khuyến nghị source-doc đổi sang
   `01-Training-Plan` và `02-Attendance-and-Class-Records`.

3. **`{Rev}` vs `{Revision}`** — §11 line 521 path dùng `REV-{Rev}`;
   §5 line 237 path dùng `REV-{Revision}`; §12 token table chỉ liệt kê
   `{Revision}`. Blueprint dùng cả 2 dạng theo đúng path canonical
   tương ứng (`Part-REV-Master/{CustomerID}/{PartNo}/REV-{Rev}/` cho
   §11; `Job-Dossiers/{YYYY}/{JobNum}-{PartNo}-REV-{Revision}/` cho §5).

4. **`{ID-Name}` placeholder** — §9 line 440 dùng cho contractors.
   Không xuất hiện trong §12 token table như một entry riêng. Giữ
   nguyên literal.

5. **Customer-Received / Tooling-Fixture-Gage / HR-Operations / Cleanroom-Records / Subcontractor-Records** — đã được spec hoá đầy đủ cấp L2/L3 tại `ANNEX-139` (rev 2026-05-09) cho 6 thư viện IP/restricted; blueprint nay đã tạo full cấu trúc thư mục đúng theo §2-§8 của ANNEX-139. Note cũ "chỉ tạo root với .gitkeep" đã được thay thế bởi đợt rebuild 2026-05-16.

encoded in the script. Re-running is the only correct way to amend the
blueprint — never edit folders by hand.
