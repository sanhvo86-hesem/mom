Dùng khi thiết lập và vận hành vòng lặp tài liệu số giữa máy local đang đồng bộ qua ShareFile, kho Git trung tâm và portal đang chạy trên cPanel. WI này là hướng dẫn thực thi bắt buộc cho người mới bắt đầu, Document Owner, QMS Engineer, IT Administrator và người được giao quyền đồng bộ trên portal.

WI-107 — Quy trình ShareFile, Git và cPanel cho vòng lặp kiểm soát tài liệu khép kín

KẾT QUẢ BẮT BUỘC: một thay đổi tài liệu phải đi theo một vòng lặp khép kín có dấu vết rõ ràng từ nơi soạn thảo local trên ShareFile, qua Git, xuống cPanel để trở thành bản point-of-use, rồi khi có chỉnh sửa khẩn trên cPanel phải quay ngược về Git và trở lại local. Không được để tồn tại bản cPanel-only, local-only hoặc bản nhớ cá nhân làm nguồn phát hành hiệu lực.

1. Nguyên tắc điều hành bắt buộc

Hướng chuẩn để phát hành

Chuẩn ShareFile local → GitHub → cPanel. Đây là hướng mặc định cho mọi thay đổi đã chuẩn bị đầy đủ, có DCR và review.

Hướng ngoại lệ để cứu live system

Ngoại lệ cPanel → GitHub → ShareFile local. Chỉ dùng khi cần sửa trực tiếp trên portal/cPanel để bảo vệ vận hành, sửa liên kết, sửa lỗi hiển thị hoặc xử lý sự cố point-of-use.

Không đồng bộ file tạm

Loại trừ Các file runtime như sessions, ratelimit, scan_cache.json, php_error.log, form-workflow và users.json không được xem là thay đổi tài liệu để phát hành.

Quy tắc một người tích hợp: thư mục Git working copy nằm trong ShareFile chỉ có một người tích hợp chính được quyền chỉnh sửa trực tiếp cùng một thời điểm. Người khác gửi yêu cầu thay đổi qua DCR/review pack hoặc làm việc trên bản clone riêng của mình, không cùng ghi vào một thư mục .git được ShareFile đồng bộ.

3. Thiết lập lần đầu cho người mới

3.1 Chuẩn bị ShareFile root

  • Tạo một root duy nhất trong vùng Shared Folders của ShareFile theo bản thiết kế tại ANNEX-116.
  • QMS Engineer và IT Administrator giữ quyền Admin ở root; Document Owner chỉ được quyền theo vai trò ở từng thư mục con.
  • Không dùng Personal Folders cho bản phát hành hiệu lực, không đặt thư mục Git working copy trong vùng cá nhân.

3.2 Chuẩn bị local working copy trong ShareFile

Dùng thư mục local đã được ShareFile đồng bộ, ví dụ C:\ShareFile\HESEM-QMS-Control\01-Controlled-Working-Repos\qms.hesem.com.vn.

cd C:\ShareFile\HESEM-QMS-Control\01-Controlled-Working-Repos
git clone https://github.com/<account>/<repo>.git qms.hesem.com.vn
cd qms.hesem.com.vn
git branch --show-current
git remote -v
  • Nhánh chuẩn để vận hành live là main. Không đổi nhánh trên cPanel nếu chưa có chấp thuận của IT Administrator và QMS Engineer.
  • Kiểm tra git remote -v để chắc local đang trỏ đúng repo đang được cPanel dùng làm origin.

3.3 Chuẩn bị cPanel

cd /home/<cpanel-user>/qms.hesem.com.vn
git rev-parse --is-inside-work-tree
git config user.name "<cpanel sync user>"
git config user.email "<cpanel-sync>@local.invalid"
ssh-keygen -t ed25519 -C "<cpanel-user>@cpanel" -f ~/.ssh/id_ed25519 -N ""
cat ~/.ssh/id_ed25519.pub
ssh -T git@github.com
git remote set-url origin git@github.com:<account>/<repo>.git
git pull --ff-only origin main
  • Public key ở bước trên phải được thêm vào tài khoản GitHub đang sở hữu repo.
  • Portal Admin chỉ được dùng nút đồng bộ khi cPanel repo đã qua bước SSH key test thành công.
  • Nếu cPanel không nhìn thấy repo, phải sửa ở repo path thật đang chạy portal, không sửa ở bản sao ngoài repo.

4. Chu kỳ chuẩn: ShareFile local → GitHub → cPanel

Bước 1. Mở DCR theo FRM-102, quyết định đây là thay đổi minor hay major, và chốt tài liệu liên đới cần rà soát.
Bước 2. Trước khi sửa, kéo local về mới nhất để tránh divergence. 
cd C:\ShareFile\HESEM-QMS-Control\01-Controlled-Working-Repos\qms.hesem.com.vn
git stash push -m "temp-before-pull"
git pull origin main
git stash pop
Bước 3. Chỉnh sửa nội dung ngay trong working copy nằm dưới ShareFile. Không sửa trong thư mục export cá nhân. Không đổi tên file tùy ý ngoài rule đặt tên chuẩn.
Bước 4. Tự kiểm tra: liên kết nội bộ còn mở được, mã tài liệu đúng, version label đúng, revision note rõ, link FRM/WI/ANNEX/SOP không chết.
Bước 5. Đưa bản dự thảo và peer review log vào thư mục review pack trong ShareFile theo ANNEX-116.
Bước 6. Commit và push local lên GitHub. 
git status --short
git add <file-1> <file-2>
git commit -m "docs: <noi-dung-thay-doi>"
git push origin main
Bước 7. Trên cPanel dùng một trong hai cách:
  • Cách có giao diện: đăng nhập portal Admin và bấm Pull to Portal.
  • Cách terminal: chạy git pull --ff-only origin main trong repo path trên cPanel.
Bước 8. Xác nhận point-of-use sau pull: mở đúng tài liệu live trên portal, kiểm tra version, link và nội dung ở đúng nơi dùng.
Bước 9. Lưu bằng chứng phát hành vào ShareFile: DCR, peer review, commit hash, ngày pull trên cPanel, ảnh/smoke test hoặc check list triển khai.
Bước 10. Cập nhật sổ đăng ký và thông báo triển khai theo SOP-101.

5. Chu kỳ ngoại lệ: cPanel → GitHub → ShareFile local

Chỉ dùng khi có lý do rõ: sửa lỗi live point-of-use, sửa liên kết hỏng, sửa chính tả/hiển thị gây hiểu sai, hoặc xử lý gián đoạn mà không thể chờ vòng local chuẩn.
Bước 1. Chỉ sửa trong repo thật đang chạy portal. Không sửa ở thư mục public_html hay bản sao ngoài repo.
Bước 2. Sau khi sửa trên portal/cPanel, bấm Push to Git. Nút này sẽ commit các thay đổi meaningful và tự bỏ qua file runtime.
Bước 3. Về local working copy trong ShareFile và kéo thay đổi mới nhất về. 
cd C:\ShareFile\HESEM-QMS-Control\01-Controlled-Working-Repos\qms.hesem.com.vn
git pull origin main
Bước 4. Đối chiếu local với nội dung đang live trên cPanel, bổ sung release evidence vào ShareFile và cập nhật DCR nếu đây là thay đổi có kiểm soát.
Bước 5. Nếu thay đổi cPanel là thay đổi logic/quy trình chứ không chỉ là hotfix hiển thị, Document Owner phải chuẩn hóa lại revision note và hoàn tất review/approval ngay trong ngày làm việc kế tiếp.
Lưu ý quan trọng về nút Pull: khi bấm Pull to Portal, hệ thống hiện tại sẽ tự dọn runtime noise, thử pre-sync các thay đổi meaningful trên cPanel trước, rồi mới chạy pull --ff-only. Tuy nhiên về vận hành chuẩn, sau khi vừa sửa trên cPanel bạn vẫn nên bấm Push to Git rõ ràng trước để dấu vết phát hành minh bạch hơn.

9. Điều cấm bắt buộc

  • Không sửa tài liệu live ở thư mục ngoài repo rồi mong git pull tự đồng bộ ngược.
  • Không dùng cùng một working copy Git trong ShareFile cho nhiều người cùng ghi đồng thời.
  • Không để thay đổi cPanel không được đẩy về Git và local quá một ca làm việc.
  • Không bỏ qua DCR/release evidence chỉ vì đã có commit hash.
  • Không xem file runtime là chứng cứ thay đổi tài liệu.
