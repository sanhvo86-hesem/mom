ANNEX-113 — dashboard governance ở mức vận hành

ANNEX này không chỉ quy định “dashboard nào nên có”, mà khóa toàn bộ đường đi từ source-of-record → dataset → semantic model → report/workspace → pack review → quyết định điều hành. Một dashboard production chỉ được coi là dùng được khi có owner, boundary quyền, refresh rule, failure playbook, rollback path và release evidence rõ.

PHẢI
Dashboard production chỉ publish từ dataset / semantic model có business owner, data owner, environment và release evidence rõ.
PHẢI
Mọi dashboard dùng để ra quyết định ở T1/T2/T3/T4 phải chỉ rõ source-of-record, freeze-date, thời điểm refresh gần nhất và cách xử lý khi refresh fail.
PHẢI
Quyền Publish / Admin / Build không được cấp kiểu “cho tiện”; phải bám ANNEX-101, ANNEX-102 và ANNEX-120.
PHẢI
Dùng pipeline DEV → UAT → PROD và rollback nhanh thay vì sửa thẳng report production trong giờ họp hoặc cuối kỳ.
Boundary của ANNEX
ANNEX-113 khóa lớp kiểm soát số. Định nghĩa KPI, formula, dimension, cut-off logic và SoR precedence chi tiết nằm tại ANNEX-110. Cách tổ chức dashboard pack, ghi chú điều hành, exception note và management-bộ hồ sơ rà soát (review pack) nằm tại WI-901 và SOP-902.

1. Mục đích, phạm vi và boundary

Mục đích

Giữ cho dashboard production là một lớp hiển thị đáng tin để điều hành, không trở thành nguồn tạo số “mạnh ai nấy sửa”. Khi dữ liệu số lỗi hoặc chậm, ANNEX này buộc tổ chức quay về SoR, khóa exception note và backfill có kiểm soát.

Phạm vi

Áp dụng cho Power BI, portal scorecards, exported review packs, monthly / quarterly KPI packs, semantic model/dataset, automation liên quan refresh và các bảng điều hành dùng cho T1/T2/T3/T4, xem xét của lãnh đạo (management review), customer review hoặc internal governance packs.

Không thuộc ANNEX nàyKPI definition sâu, process capture rule và dataset field dictionary đã được khóa tại ANNEX-110 và các annex kỹ thuật liên quan.
Điểm giaoANNEX-113 bắt đầu từ lúc một chỉ số được đưa vào dashboard pack, được refresh, được publish, được review và/hoặc được dùng để quyết định.
Output bắt buộcDashboard register, release pack, refresh evidence, nhật ký sự cố (incident log), exception note, backfill close-out và access review evidence.

Release class A — cosmetic / ghi chú điều hành

Đổi nhãn, format, layout, grouping hoặc ghi chú điều hành text không thay nghĩa KPI. Vẫn cần change note và người phụ trách.

Release class B — logic / threshold

Đổi formula, threshold, cut-off, precedence, drill-down keys hoặc mapping có thể ảnh hưởng quyết định. Bắt buộc test, owner sign-off và snapshot trước/sau.

Release class C — source / environment / security

Đổi nguồn, connector, workspace, permission boundary, refresh pipeline hoặc service principal. Bắt buộc rollback path, approver cao hơn và review hậu phát hành.

8. KPI của chính hệ dashboard governance

% refresh đúng cửa sổ

Đo độ ổn định vận hành dashboard; tách theo dashboard critical vs non-critical.

% release có đủ pack

Đo kỷ luật change control trước khi publish lên production.

MTTR của incident dữ liệu

Đo tốc độ khôi phục khi refresh fail, mapping sai hoặc permission lỗi.

% privileged / publish access review đúng hạn

Đo chất lượng RBAC, least privilege và closure của quarterly review.
