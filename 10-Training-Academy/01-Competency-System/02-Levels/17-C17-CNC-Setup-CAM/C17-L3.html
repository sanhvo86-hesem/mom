C17 — Năng lực kỹ thuật theo vị trí (CNC/Setup/CAM) — Level 3

Level 3 — CAM/NC: thiết kế toolpath, mô phỏng, phát hành chương trình có kiểm soát. Level 3 yêu cầu làm chủ CAM/NC theo QMS: từ drawing/spec → toolpath → mô phỏng → post → release → prove-out support. Mục tiêu là chương trình an toàn, ổn định và truy xuất được mọi thay đổi.

2) Mục đích mô-đun & bối cảnh vận hành

  • CAM/NC không chỉ là “ra đường chạy dao”. Đây là vị trí quyết định rủi ro crash và rủi ro chất lượng.
  • Level 3 chuyển CAM/NC sang chế độ kiểm soát: revision set → mô phỏng → peer review → release pack → kiểm soát thay đổi.
  • Bám quy trình: PROC-ENG-002 Phát hành kỹ thuật • PROC-CNC-002 Program/Setup Release Control.

3) Vị trí trong chuỗi E2E + luồng bàn giao (đầu vào/đầu ra) + điểm chặn

Mô-đun này bám theo các quy trình nội bộ liên quan và yêu cầu “không có bằng chứng = không qua cổng kiểm soát”. Link tham chiếu: WI-OPS-001 Quality Gates • WI-OPS-008 Hồ sơ công việc (job dossier) & Bộ bằng chứng (evidence pack).

4) Điểm đứt gãy (tối thiểu 10 điểm gãy)

  • Dùng sai phiên bản (wrong revision) set → gia công sai yêu cầu khách hàng.
  • Không kiểm clearance/fixture trong mô phỏng → crash tại máy.
  • Sai postprocessor hoặc sai đầu ra unit → chương trình chạy sai hành trình.
  • Thiếu kiểm travel limit → overtravel, crash.
  • Không chuẩn hóa tên file/version → operator nạp nhầm.
  • Không có peer review → lỗi chủ quan lọt.
  • Release pack thiếu setup sheet/tool list → setup làm theo kinh nghiệm, rủi ro cao.
  • Không xác định CTQ/inspection points → QC đo thiếu, lỗi lọt.
  • Sửa chương trình trực tiếp tại máy mà không cập nhật SSOT → mất kiểm soát.
  • Không quản trị thay đổi (change request) → không truy xuất nguyên nhân khi lỗi xảy ra.
  • Không archive phiên bản cũ → đánh giá không truy xuất được lịch sử.
  • Tối ưu cycle time mà bỏ qua risk (dao/độ cứng/biến dạng) → lỗi chất lượng.

6) Fail-fast rules (STOP/HOLD/BLOCK)

  • BLOCK Khi chưa chạy mô phỏng hoặc chưa có peer review log.
  • HOLD Khi release pack thiếu setup sheet/tool list/CTQ/inspection points.
  • STOP Khi phát hiện operator đang dùng file rời/USB cá nhân.
  • BLOCK Khi thay đổi chương trình không có change request và không update SSOT.

7) Phản ứng 60 giây

  1. DỪNG phát hành và khóa release pack hiện tại (không gửi file rời).
  2. XÁC NHẬN revision set với ENG/QA; cập nhật baseline khi có thay đổi.
  3. CHẠY lại mô phỏng và cập nhật peer review log cho bản mới.
  4. PHÁT HÀNH lại release pack qua SSOT; thông báo cho setup/operator dùng đúng bản.
  5. ARCHIVE bản cũ và ghi rõ lý do thay đổi (change request + bằng chứng).

10) Đánh giá

Bài kiểm tra nhanh (≥ 20 câu)

  1. Revision set là gì và vì sao phải khóa trước khi CAM?
  2. Mô phỏng phải kiểm những rủi ro nào tối thiểu?
  3. Vì sao travel limit là điểm chặn crash?
  4. Sai postprocessor tạo ra lỗi gì?
  5. Naming rule giúp chặn rủi ro gì?
  6. Peer review log dùng để làm gì?
  7. Release pack bắt buộc gồm những thành phần nào?
  8. Setup sheet thiếu dẫn tới rủi ro gì ở hiện trường?
  9. CTQ phải được gắn vào release pack như thế nào?
  10. FRM-QA-016 khác gì FRM-QMS-007?
  11. Vì sao phải chặn việc sửa chương trình tại máy mà không update SSOT?
  12. Change request bắt buộc trong các trường hợp nào?
  13. Archive version giúp trả lời câu hỏi đánh giá nào?
  14. Tối ưu cycle time phải chặn rủi ro nào trước?
  15. Khi phát hiện collision trong mô phỏng, bước xử lý bắt buộc là gì?
  16. Khi drawing đổi revision, CAM phải làm gì ngay lập tức?
  17. Nêu 3 lỗi hay gặp do đặt tên file không chuẩn.
  18. Vì sao phải xác định inspection points trước prove-out?
  19. Khi setup phản hồi cần thay dao, CAM xử lý qua cơ chế nào?
  20. Khi release pack phát hành, ai phải nhận thông báo?
  21. Nêu 2 bằng chứng bắt buộc phải lưu trong SSOT sau khi release program.
  22. Nêu 3 lỗi đánh giá hay gặp liên quan quản trị chương trình CNC.

Scenario (≥ 3 tình huống thật)

  1. Scenario 1 — Mô phỏng phát hiện collision
    Khi mô phỏng phát hiện dao chạm đồ gá. Hãy mô tả quy trình chỉnh sửa: toolpath → mô phỏng lại → peer review → release pack.
  2. Scenario 2 — Operator dùng file rời
    Khi phát hiện operator đang dùng file NC từ USB. Hãy thực hiện fail-fast (STOP), thu hồi file rời và phát hành lại qua SSOT.
  3. Scenario 3 — Tối ưu cycle time gây drift
    Sau tối ưu cycle time, kích thước drift và bề mặt xấu. Hãy mô tả cách mở change request, cập nhật control plan và xác minh trước khi phát hành rộng.

Performance test

Bài test tay nghề đánh giá khả năng phát hành chương trình có kiểm soát theo QMS.

  • Từ 01 bộ hồ sơ chuẩn mẫu: tạo toolpath và chạy mô phỏng; lưu screenshot bằng chứng.
  • Soạn release pack: NC program + setup sheet + CTQ/inspection points + release bảng kiểm.
  • Thực hiện peer review (FRM-QMS-007) và đóng gói link SSOT.
  • Mô phỏng thay đổi: mở change request, cập nhật version, archive bản cũ.

11) ĐẠT/KHÔNG ĐẠT criteria

đạt khiKhông đạt khi
  • Release pack đầy đủ và truy xuất được trong SSOT; naming chuẩn.
  • Có mô phỏng bằng chứng và peer review log trước phát hành.
  • Mọi thay đổi có change request và có checksum / manifest.
  • CTQ/inspection points rõ ràng, hỗ trợ prove-out.
  • Phát hành khi chưa mô phỏng hoặc chưa peer review.
  • Release pack thiếu setup sheet hoặc thiếu CTQ/inspection points.
  • Cho phép sửa chương trình tại máy mà không update SSOT.
  • Không checksum / manifest hoặc không ghi lý do thay đổi.

12) MRR tối thiểu (hồ sơ bắt buộc + naming + nơi lưu)

  • Bộ hồ sơ chuẩn link trong SSOT

  • FRM-ENG-002 Setup Sheet
  • FRM-QA-016 Release Bảng kiểm
  • FRM-ENG-003 Change Request
  • Mô phỏng bằng chứng (screenshot) trong SSOT
  • Archive version hồ sơ

SSOT: M365/SharePoint → CNC-IMS-HUB → JobDossier → JobNum → EvidencePack. Quy tắc đặt tên bám theo WI-OPS-008.

13) Common đánh giá phát hiện & phòng ngừa

  • Không chứng minh mô phỏng/peer review trước phát hành.
  • Release pack thiếu setup sheet hoặc thiếu plan đo CTQ.
  • Chương trình chạy tại máy không khớp SSOT version.
  • Không có change request cho thay đổi chương trình.
  • Không checksum / manifest cũ; DCR hồ sơ không truy xuất.
  • Naming file không chuẩn, operator nạp nhầm.
  • Inspection points không rõ, QC đo thiếu.
  • Cycle time tối ưu gây lỗi nhưng không có verification.
