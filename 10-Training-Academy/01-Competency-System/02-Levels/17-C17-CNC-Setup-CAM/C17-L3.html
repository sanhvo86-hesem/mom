C17 — Competence kỹ thuật theo vị trí (CNC/Setup/CAM) — Level 3

Level 3 — CAM/NC: thiết kế đường chạy dao, mô phỏng, release chương trình có kiểm soát. Level 3 requirement làm chủ CAM/NC theo QMS: từ drawing/spec → đường chạy dao → mô phỏng → post → release → prove-out support. Mục tiêu là chương trình an toàn, ổn định và traceability được mọi thay đổi.

2) Mục đích mô-đun & bối cảnh operation

  • CAM/NC không chỉ là “ra đường chạy dao”. Đây là vị trí quyết định rủi ro crash và rủi ro quality.
  • Level 3 chuyển CAM/NC sang chế độ kiểm soát: revision set → mô phỏng → review đồng cấp → gói release → change control.
  • Bám process: PROC-ENG-002 Release kỹ thuật • PROC-CNC-002 Program/Setup Release Control.

3) Vị trí trong chuỗi E2E + luồng bàn giao (input/output) + hold point

4) Điểm đứt gãy (tối thiểu 10 điểm gãy)

  • Dùng sai revision (wrong revision) set → gia công sai requirement customer.
  • Không kiểm clearance/fixture trong mô phỏng → crash tại máy.
  • Sai postprocessor hoặc sai output unit → chương trình chạy sai hành trình.
  • Thiếu inspectionvel limit → overtravel, crash.
  • Không chuẩn hóa tên file/version → operator nạp nhầm.
  • Không có review đồng cấp → lỗi chủ quan lọt.
  • Gói release thiếu setup sheet/tool list → setup làm theo kinh nghiệm, rủi ro cao.
  • Không xác định CTQ/inspection points → QC đo thiếu, lỗi lọt.
  • Sửa chương trình trực tiếp tại máy mà không cập nhật SSOT → mất kiểm soát.
  • Không quản trị thay đổi (requirement thay đổi) → không traceability nguyên nhân khi lỗi xảy ra.
  • Không archive revision cũ → đánh giá không traceability được lịch sử.
  • Tối ưu cycle time mà bỏ qua risk (dao/độ cứng/biến dạng) → lỗi quality.

6) Fail-fast rules (STOP/HOLD/BLOCK)

  • BLOCK Khi chưa chạy mô phỏng hoặc chưa có Peer Review Log.
  • HOLD Khi gói release thiếu setup sheet/tool list/CTQ/inspection points.
  • STOP Khi phát hiện operator đang dùng file rời/USB cá nhân.
  • BLOCK Khi thay đổi chương trình không có requirement thay đổi và không update SSOT.

7) Phản ứng 60 giây

  1. DỪNG release và khóa gói release hiện tại (không gửi file rời).
  2. XÁC NHẬN revision set với ENG/QA; cập nhật baseline khi có thay đổi.
  3. CHẠY lại mô phỏng và cập nhật Peer Review Log cho bản mới.
  4. PHÁT HÀNH lại gói release qua SSOT; thông báo cho setup/operator dùng đúng bản.
  5. ARCHIVE bản cũ và ghi rõ lý do thay đổi (requirement thay đổi + evidence).

10) Đánh giá

Bài inspection nhanh (≥ 20 câu)

  1. Revision set là gì và vì sao phải khóa trước khi CAM?
  2. Mô phỏng phải kiểm những rủi ro nào tối thiểu?
  3. Vì sao travel limit là hold point crash?
  4. Sai postprocessor tạo ra lỗi gì?
  5. Naming rule giúp chặn rủi ro gì?
  6. Peer Review Log dùng để làm gì?
  7. Gói release bắt buộc gồm những thành phần nào?
  8. Setup sheet thiếu dẫn tới rủi ro gì ở hiện trường?
  9. CTQ phải được gắn vào gói release như thế nào?
  10. FRM-QA-016 khác gì FRM-QMS-007?
  11. Vì sao phải chặn việc sửa chương trình tại máy mà không update SSOT?
  12. Requirement thay đổi bắt buộc trong các trường hợp nào?
  13. Archive version giúp trả lời câu hỏi đánh giá nào?
  14. Tối ưu cycle time phải chặn rủi ro nào trước?
  15. Khi phát hiện collision trong mô phỏng, bước xử lý bắt buộc là gì?
  16. Khi drawing đổi revision, CAM phải làm gì ngay lập tức?
  17. Nêu 3 lỗi hay gặp do đặt tên file không chuẩn.
  18. Vì sao phải xác định inspection points trước prove-out?
  19. Khi setup phản hồi cần thay dao, CAM xử lý qua cơ chế nào?
  20. Khi gói release release, ai phải nhận thông báo?
  21. Nêu 2 evidence bắt buộc phải lưu trong SSOT sau khi release program.
  22. Nêu 3 lỗi đánh giá hay gặp liên quan quản trị chương trình CNC.

Scenario (≥ 3 tình huống thật)

  1. Scenario 1 — Mô phỏng phát hiện collision
    Khi mô phỏng phát hiện dao chạm đồ gá. Hãy mô tả process chỉnh sửa: đường chạy dao → mô phỏng lại → review đồng cấp → gói release.
  2. Scenario 2 — Operator dùng file rời
    Khi phát hiện operator đang dùng file NC từ USB. Hãy thực hiện fail-fast (STOP), recall file rời và release lại qua SSOT.
  3. Scenario 3 — Tối ưu cycle time gây drift
    Sau tối ưu cycle time, kích thước drift và bề mặt xấu. Hãy mô tả cách mở requirement thay đổi, cập nhật control plan và xác minh trước khi release rộng.

Performance test

Bài test tay nghề đánh giá khả năng release chương trình có kiểm soát theo QMS.

  • Từ 01 bộ record chuẩn mẫu: tạo đường chạy dao và chạy mô phỏng; lưu screenshot evidence.
  • Soạn gói release: NC program + setup sheet + CTQ/inspection points + release checklist.
  • Thực hiện review đồng cấp (FRM-QMS-007) và packaging link SSOT.
  • Mô phỏng thay đổi: mở requirement thay đổi, cập nhật version, archive bản cũ.

11) ĐẠT/KHÔNG ĐẠT criteria

đạt khiKhông đạt khi
  • Gói release đầy đủ và traceability được trong SSOT; naming chuẩn.
  • Có mô phỏng evidence và Peer Review Log trước release.
  • Mọi thay đổi có requirement thay đổi và có checksum / bảng kê.
  • CTQ/inspection points rõ ràng, hỗ trợ prove-out.
  • Release khi chưa mô phỏng hoặc chưa review đồng cấp.
  • Gói release thiếu setup sheet hoặc thiếu CTQ/inspection points.
  • Cho phép sửa chương trình tại máy mà không update SSOT.
  • Không checksum / bảng kê hoặc không ghi lý do thay đổi.

13) Common đánh giá phát hiện & phòng ngừa

  • Không chứng minh mô phỏng/review đồng cấp trước release.
  • Gói release thiếu setup sheet hoặc thiếu plan đo CTQ.
  • Chương trình chạy tại máy không khớp SSOT version.
  • Không có requirement thay đổi cho thay đổi chương trình.
  • Không checksum / bảng kê cũ; DCR record không traceability.
  • Naming file không chuẩn, operator nạp nhầm.
  • Inspection points không rõ, QC đo thiếu.
  • Cycle time tối ưu gây lỗi nhưng không có xác minh.
